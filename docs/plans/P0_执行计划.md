---

## 1. 执行原则

P0 执行时遵守以下原则：

1. **先固定边界，再写文件。**
2. **先搭骨架，不做复杂功能。**
3. **先定义规则，不写重脚本。**
4. **先建空壳 skills，不追求自动执行。**
5. **先确保 Codex 可理解，再进入 P1 闭环。**
6. **所有目录、模板、配置都服务于后续证据追溯和多对多映射。**
7. **完成 P0 验收后立即暂停，不继续扩展到 P1/P2/P3 功能。**

## P0-05：创建 config 文件空壳

### 目标

建立最小配置层，后续 P1/P2 可以逐步填充。

### 具体动作

创建：

```text
config/research_config.yaml
config/segment_taxonomy.yaml
config/source_registry.yaml
config/metric_definitions.yaml
config/scoring_frameworks.yaml
config/watchlist.yaml
```

### 推荐最小内容

`research_config.yaml`：

```yaml
project:
  name: a-share-research-os
  language: zh-CN
  date_format: YYYY-MM-DD

evidence:
  id_pattern: evi_<source_type>_<entity>_<date>_<short_hash>
  confidence_labels: [high, medium, low]
  status_labels: [fresh, stale, superseded, contradicted, low_confidence]

claims:
  id_pattern: claim_<entity_type>_<entity_id>_<date>_<seq>
  claim_types: [fact, estimate, inference, management_comment, analyst_view, opinion]

reports:
  status_labels: [current, needs_refresh, outdated, archived]
```

`segment_taxonomy.yaml`：

```yaml
segments: []
schema:
  segment_id: string
  name_cn: string
  name_en: string
  aliases: list
  definition: string
  scope_in: list
  scope_out: list
  parent_theme: string
  industry_chain_role: string
  status: active|watch|archived
```

`source_registry.yaml`：

```yaml
source_types:
  annual_report:
    reliability_rank: A
    use_case: company_fact
  announcement:
    reliability_rank: A
    use_case: company_event
  exchange_data:
    reliability_rank: A
    use_case: market_or_disclosure
  regulator_statistics:
    reliability_rank: A
    use_case: macro_or_industry_fact
  industry_report:
    reliability_rank: B
    use_case: industry_estimate
  brokerage_report:
    reliability_rank: B
    use_case: analyst_view
  news:
    reliability_rank: C
    use_case: event_tracking
  expert_call:
    reliability_rank: C
    use_case: qualitative_check
```

### 产出文件

```text
config/*.yaml
```

### 依赖

P0-04。

### 验收标准

- 6 个配置文件都存在。
- 每个文件都有 schema 或示例结构。
- 不填入未经证据支持的真实研究结论。

## P0-08：编写 skills 空壳

### 目标

给每个 skill 建立明确边界，防止 Codex 混用任务。

### 具体动作

为每个 `SKILL.md` 写入统一结构：

```md
---
name: <skill-name>
description: <何时使用。必须清晰描述触发场景和不要使用的场景。>
---

# <Skill Name>

## Goal

## When to use

## Inputs

## Responsibilities

## Out of scope

## Outputs

## Workflow

## Guardrails

## Quality checklist
```

### 每个 skill 的关键边界

| Skill | 必须写清楚 |
|---|---|
| `evidence-ingest` | 只负责证据导入、归档、登记，不负责投资结论 |
| `segment-research` | 负责细分定义、产业链、公司池、评分，不负责单一个股深度 |
| `company-universe` | 负责找公司池和暴露层级，不负责估值 |
| `segment-company-mapping` | 负责多对多关系和暴露字段，不负责写完整报告 |
| `stock-deep-dive` | 负责个股深度，不负责细分横向排序 |
| `compare-segments` | 负责多个细分横向比较，不给买卖建议 |
| `compare-stocks` | 负责个股横向比较，不把评分等同交易信号 |
| `refresh-research` | 只输出变化，不静默重写全部报告 |
| `quality-review` | 检查证据、口径、反证、事实/推断区分 |
| `memo-writer` | 生成备忘录和观察清单，不输出交易指令 |

### 产出文件

```text
.agents/skills/*/SKILL.md
```

### 依赖

P0-07。

### 验收标准

- 每个 `SKILL.md` 都有 frontmatter。
- 每个 description 都能帮助 Codex 判断何时使用。
- 每个 skill 都有 out of scope。
- 每个 skill 都有 guardrails。
- 没有 skill 在 P0 阶段要求复杂自动化。

### 人工确认

必须确认。

## P0-11：定义质量检查规则

### 目标

建立最小质量闸门，防止 P1 生成不可追溯内容。

### 具体动作

在 `quality-review/SKILL.md` 中写入检查项：

```text
1. 是否所有关键结论都有 evidence_id 或 claim_id。
2. 是否混淆事实、估计、推断、观点。
3. 是否把管理层表述当成事实。
4. 是否把券商预测当成事实。
5. 是否标记缺失数据。
6. 是否列出反证和不确定性。
7. 是否说明指标口径、单位和周期。
8. 是否存在过期证据。
9. 是否有更新日志要求。
10. 是否避免买卖建议。
```

### 可选产出

```text
templates/quality_review_checklist.md
```

P0 默认不强制创建，除非你希望质量检查独立成模板。

### 产出文件

```text
.agents/skills/quality-review/SKILL.md
AGENTS.md
templates/*.md
```

### 依赖

P0-02、P0-06、P0-08。

### 验收标准

## 7. P0 风险与控制

| 风险 | 表现 | 控制方式 |
|---|---|---|
| 过度工程化 | P0 开始写复杂数据库、抓取器、估值模型 | 严格执行 P0 非目标 |
| 目录泛滥 | 建太多目录但没有使用规则 | 目录必须能对应后续 P1/P2/P3 产物 |
| skill 边界不清 | 所有 skill 都在写报告 | 每个 skill 必须有 out of scope |
| 证据规则不足 | 报告模板没有 evidence_id | 模板和 AGENTS 同时约束 |
| 多对多关系缺失 | 公司只能归属一个细分 | 定义 `segment_company_exposure` |
| 把评分当交易信号 | 输出买卖建议 | AGENTS 和 memo-writer 明确禁止 |
| 更新机制提前复杂化 | P0 做批量刷新 | P0 只定义 refresh 输出格式 |
| 人工确认不足 | 后续反复改目录 | P0-00、P0-02、P0-06、P0-08 必须确认 |

## 8. P0 完成定义

P0 完成不是指“系统已经能自动研究股票”，而是指：

```text
1. Codex 能理解项目定位。
2. 工作区有稳定目录。
3. 原始证据、处理数据、报告、决策记录有固定位置。
4. 10 个 skills 的边界清楚。
5. 配置文件和模板可以支撑 P1。
6. 质量纪律能防止无证据结论。
7. 已明确 P0 不继续扩功能。
```

## 9. P1 入口建议

P0 完成后，P1 的第一步不是扩展功能，而是选一个细分方向跑最小闭环。

P1 首个任务建议为：

```text
$segment-research 调研“AI服务器液冷”，深度=standard。
要求：
1. 明确细分定义和边界；
2. 找出 A 股相关公司池；
3. 区分收入暴露、产品暴露、技术储备、市场叙事；
4. 输出 scorecard；
5. 输出 evidence_map；
6. 所有关键结论必须引用 evidence_id 或标记 TODO。
```

P1 暂时只做一个细分和 1-2 家公司，验证模板、证据、skills、目录、质量检查是否真的可用。

## 10. 最终执行结论

P0 的执行重点是：

> **让工作区“不会乱”，而不是让系统“马上很强”。**

完成 P0 后，应停止追加功能，进入 P1 的单细分最小闭环验证。
