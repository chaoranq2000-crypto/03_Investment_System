## Checkpoint and resume protocol

1. 新会话先完整读取适用 `AGENTS.md`、本合同、`START_HERE.md`、当前 phase 的权威源和脚本写入行为；运行 V-012、验证分支/基线/干净状态。
2. 从 postconditions 尚未被 Git+测试证明的最早 phase 开始；不得仅凭 START 文本跳过 phase。
3. 每阶段修改前，在 START checkpoint history 新增精确 path allowlist 和一个 next safe action。使用一个 writer；subagent 只做只读审计或互不重叠的明确文件。
4. 阶段结束先运行 targeted validation、V-008、V-009 和 scope review；把结果、remaining blockers、四事实变化写入 O-002 与 START。
5. 用显式文件路径 `git add`，包括本阶段 code/tests/artifacts/O-002/START；禁止 `git add -A`。创建 phase 指定提交，随后要求 `git status --short` 为空才进入下一阶段。
6. P5 完成后，START 设为 `running`、`last_completed_phase=P5`、`next_phase=final_validation`；执行 V-001–V-010 和 scope 审查。若通过，创建 control commit `chore(v1): record final validation checkpoint`，普通 push 并等待该 exact HEAD 的 CI。
7. 首次 exact-head CI 全部成功后，使用 skill checkpoint 将 START 改为 `complete`，创建 `chore(v1): mark verified candidate complete`，再次普通 push，并等待 completion-only exact HEAD 的标准 CI/实际触发 workflow。该提交只允许修改 START。最终 CI 成功后不再改文件；最终 URL、remote SHA 和四事实只写入外部 final handoff，O-002/O-005 不声称包含 completion commit 的 CI receipt。
8. 若 completion-only CI 是瞬时基础设施失败，可对同一 HEAD 最多重跑两次；真实测试失败不得重跑掩盖，必须恢复为新 package amendment 或请求用户，因为 complete checkpoint 不可原位回退。
9. resume 时，包含当前 `START_HERE.md` 的 Git commit 就是 checkpoint commit。若 worktree dirty，只有全部路径属于 `next_phase` 且操作幂等/内部一致时才继续；否则停止并列出路径。

At minimum: update `START_HERE.md`, run phase validation, inspect scope, stage explicit paths, create the named checkpoint commit, and require a clean checkpoint before the next phase. On resume, start from the earliest phase whose postconditions are not proven.

- 工作树出现不属于当前 phase allowlist 的修改，或与用户/其他 agent 变更重叠。
- 任何写入将触及 raw、Night/Bundle17R 历史、旧 002837 run、main、其他工作树、CI workflow、依赖或未授权外部系统。
- 需要猜测发行人数据、reviewer 身份/authority/签名/决定、人工结论或 accepted/sample/P2 状态。
- 需要新增依赖、live 网络研究、破坏性操作、PR/main merge/force-push，或 Git push 出现 non-fast-forward。
- required validator/conda/credential/GitHub 不可用；外部 CI/网络连续三次不可用时保留 engineering checkpoint 并记录 release blocker。

- 对同一根因已经尝试三种实质不同方案仍失败。
